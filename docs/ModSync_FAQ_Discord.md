# ModSync FAQ

## Что это вообще такое

**ModSync** это Forge-мод для **Minecraft 1.20.1**, который автоматически синхронизирует клиентские файлы с сервером.

Он умеет:
- скачивать недостающие или устаревшие файлы перед входом
- сравнивать локальные файлы клиента с серверным manifest
- удалять ранее установленные через ModSync файлы, если их больше нет в manifest этого сервера
- показывать статус синхронизации в списке серверов
- раздавать файлы через встроенный HTTP-сервер

Поддерживаемые категории:
- `mods`
- `resourcepacks`
- `shaderpacks`
- `config`
- `optional_client`

---

## Какие версии поддерживаются

- **Minecraft**: `1.20.1`
- **Forge**: ветка `47.x`
- **Java**: `17`

Если клиент или сервер на другой версии, стабильная работа не гарантируется.

---

## Как работает ModSync

Коротко:

1. Игрок выбирает сервер в меню.
2. Клиент получает manifest сервера.
3. Клиент сканирует свои файлы.
4. ModSync сравнивает локальные файлы с серверными.
5. Недостающие или устаревшие файлы скачиваются.
6. Если были изменены restart-required файлы, клиенту предлагается перезапустить игру.

Дополнительно:
- сервер генерирует manifest автоматически
- клиент кеширует результаты скана и хэшей
- SHA-256 используется для проверки целостности
- загрузка идёт в несколько потоков
- при включённом `use_temp_files` файлы сначала пишутся во временный файл, а потом заменяются атомарно

---

## Что именно сервер раздаёт

По умолчанию:

- серверные моды берутся из папки:
  - `mods/`

- остальные категории берутся из:
  - `sync_repo/resourcepacks/`
  - `sync_repo/shaderpacks/`
  - `sync_repo/configs/`
  - `sync_repo/optional_client/`

Типовая структура:

```text
server/
  mods/
  config/
  sync_repo/
    resourcepacks/
    shaderpacks/
    configs/
    optional_client/
```

---

## Куда клиент ставит файлы

Категории раскладываются так:

- `MOD` -> `mods/`
- `RESOURCEPACK` -> `resourcepacks/`
- `SHADERPACK` -> `shaderpacks/`
- `CONFIG` -> `config/`
- `OPTIONAL_CLIENT` -> путь из `optional_client_target`

По умолчанию `optional_client_target = "mods"`, то есть optional client контент тоже попадёт в `mods/`, если ты это явно не поменяешь.

## Где лежат важные файлы ModSync

- Конфиг:
  - `config/modsync.toml`

- Сгенерированный manifest:
  - `config/modsync-manifest.json`

- Состояние файлов по серверам:
  - `config/modsync-state/*.json`

- Кеш хэшей файлов:
  - `config/modsync-hash-cache.json`

- Логи:
  - `logs/modsync.log`

## Какие есть основные настройки

### HTTP и раздача файлов

- `enable_http_server`
  - включает встроенный HTTP-сервер ModSync

- `server_http_port`
  - порт встроенного HTTP-сервера

- `server_http_bind`
  - адрес, на котором сервер слушает HTTP
  - обычно `0.0.0.0`

- `public_http_base_url`
  - внешний URL, если раздача идёт через reverse proxy
  - если пусто, используются прямые ссылки на `server_http_port`

### Категории синхронизации

- `enable_mods_sync`
- `enable_resourcepacks_sync`
- `enable_shaderpacks_sync`
- `enable_config_sync`
- `enable_optional_client_sync`

### Загрузки

- `download_threads`
  - число потоков загрузки

- `retry_count`
  - число повторных попыток при ошибке

- `verify_hash_after_download`
  - проверять ли SHA-256 после скачивания

- `delete_invalid_files`
  - удалять ли битые или неподходящие файлы

- `use_temp_files`
  - писать ли сначала во временный файл

### Прочее

- `optional_client_target`
  - куда класть `optional_client`

- `skip_file_extensions`
  - какие расширения не учитывать при скане

- `log_to_file`
- `log_file`

---

## Рекомендуемые настройки для большинства серверов

Базовый безопасный вариант:

```toml
enable_http_server = true
server_http_port = 8080
server_http_bind = "0.0.0.0"
public_http_base_url = ""

enable_mods_sync = true
enable_resourcepacks_sync = true
enable_shaderpacks_sync = false
enable_config_sync = true
enable_optional_client_sync = false

download_threads = 4
retry_count = 3
verify_hash_after_download = true
delete_invalid_files = true
use_temp_files = true
```

Если нужен более лёгкий режим:

- отключить `shaderpacks`, если реально не нужны
- отключить `optional_client`, если не используешь
- расширить список `skip_file_extensions`

## Какие команды есть на сервере

Все команды требуют высокий уровень прав.

### Полный reload ModSync

```text
/modsync reload
```

Что делает:
- перечитывает конфиг
- пересобирает manifest
- обновляет встроенный HTTP-сервер

### Принудительно обновить manifest

```text
/modsync manifest refresh
```

### Показать текущий статус

```text
/modsync status
```

Показывает:
- включён ли HTTP
- bind/port
- `public_http_base_url`
- сколько записей в manifest
- время последней генерации

### Изменить HTTP порт

```text
/modsync config set http_port 8080
```

### Изменить внешний base URL

```text
/modsync config set public_http_base_url https://example.com/modsync
```

## Что видит игрок в клиенте

В меню серверов ModSync показывает:

- ping
- версию сервера
- состояние синхронизации
- краткие детали по серверу

Статусы:

- `Не проверено`
  - сервер ещё не проверялся

- `Проверка`
  - идёт запрос manifest и сверка файлов

- `Синхронизировано`
  - всё совпадает

- `Нужно обновить`
  - есть недостающие или устаревшие файлы

- `Manifest недоступен`
  - manifest не удалось получить

---

## Что происходит при нажатии "Скачать моды" или "Подключиться"

Оба сценария запускают pre-join sync:

1. клиент получает manifest
2. сравнивает локальные файлы
3. скачивает нужное
4. если всё готово, даёт продолжить

Если были изменены restart-required файлы:
- появляется экран перезапуска
- показывается список обновлённых/удалённых файлов

---

## Почему ModSync может удалить файлы

Удаление происходит не просто так.

ModSync удаляет только те файлы, которые:
- были ранее записаны как управляемые этим сервером
- и больше не присутствуют в новом manifest этого же сервера

Это нужно, чтобы клиент не накапливал мусор и старые версии файлов.

Важно:
- логика синка отключена для singleplayer/local session
- случайное удаление в одиночке не должно происходить
- если раньше уже был старый state-файл, стоит проверить `config/modsync-state/`

---

## Может ли игрок зайти без ModSync

Нет, сервер теперь ожидает ModSync handshake.

Если клиент дошёл до входа, но не отправил handshake за короткое время, сервер кикнет его с сообщением:

```text
Missing required client sync mod. Install Forge and ModSync to join this server.
```

Нюанс:
- чистый клиент без Forge может быть отклонён ещё раньше стандартной логикой Forge
- но игрок без ModSync-handshake уже точно не останется на сервере

---

## Почему ModSync просит перезапустить игру

Потому что часть файлов нельзя корректно "подменить на лету".

Обычно перезапуск нужен для:
- модов
- optional client файлов, если они попали в `mods/`

Обычно не требует рестарта:
- resourcepacks
- shaderpacks
- config

Но итоговое поведение зависит от типа контента и того, куда он ставится.

---

## Что означает ошибка SHA-256

Если в логах есть что-то вроде:

```text
Hash verification failed
Invalid SHA-256
```

это значит:
- сервер отдал файл
- но его реальный хэш не совпал с тем, что записан в manifest

Обычно причины:
- файл на сервере поменяли после генерации manifest
- manifest устарел
- reverse proxy отдаёт не тот файл
- файл повредился при раздаче

Что делать:
1. выполнить `/modsync manifest refresh`
2. проверить, что файл в `mods/` или `sync_repo/` действительно тот, который должен раздаваться
3. проверить proxy и URL

---

## Что делать, если статус всё время "Проверка"

Проверь:

1. доступен ли встроенный HTTP сервер
2. совпадает ли `server_http_port`
3. корректно ли вшит скрытый HTTP порт в MOTD
4. не блокирует ли firewall HTTP-порт
5. нет ли ошибок в `logs/modsync.log`

Если используешь reverse proxy:
- проверь `public_http_base_url`
- проверь проксирование `/manifest`
- проверь проксирование `/files/...`

---

## Что делать, если сервер пишет "Manifest недоступен"

Проверь:

- запущен ли HTTP-сервер ModSync
- свободен ли `server_http_port`
- правильный ли bind
- доступен ли сервер по IP/домену
- не режет ли CDN или proxy запросы к `/manifest`

Полезная проверка:

```text
http://<server-ip>:<port>/manifest
```

Если manifest не открывается из браузера или через curl, клиент тоже его не получит.

---

## Что делать, если игрок застрял на "нет активных загрузок"

Смотри лог окна прогресса и `logs/modsync.log`.

Чаще всего причина:
- хэш не совпал
- сервер недоступен по HTTP
- файл не найден
- proxy режет загрузку

Если загрузки закончились ошибкой, это не значит, что синк успешно завершился.

---

## Почему ModSync может нагружать диск и CPU

Потому что ему нужно:
- сканировать клиентские и серверные папки
- считать SHA-256
- сравнивать большие списки файлов

Сейчас в моде уже есть оптимизации:
- увеличен TTL локального кэша scan snapshot
- убраны лишние повторные клиентские сканы
- добавлен persistent cache хэшей
- ускорен hex encoder

Если файлов очень много, первая генерация всё равно будет ощутимо тяжелее повторных запусков.

---

## Как ускорить работу

### Для сервера

- не хранить лишний мусор в `mods/` и `sync_repo/`
- отключить ненужные категории
- не держать архивы, бэкапы и временные файлы рядом с раздаваемыми файлами
- расширить `skip_file_extensions`

Пример:

```toml
skip_file_extensions = [".txt", ".log", ".tmp", ".bak", ".disabled", ".old", ".backup", ".orig", ".cache"]
```

### Для клиента

- не смешивать разные большие сборки в одной папке без необходимости
- не класть мусор в категории, которые ModSync сканирует

---

## Reverse proxy: когда нужен

Нужен, если:
- ты не хочешь открывать прямой `server_http_port`
- хочешь раздавать файлы через домен
- хочешь спрятать внутреннюю схему раздачи

Настройка:

```toml
public_http_base_url = "https://your-domain.example/modsync"
```

Proxy должен корректно пропускать:
- `/manifest`
- `/files/...`

---

## Безопасность

В моде уже есть несколько защит:

- блокируется path traversal при работе с путями
- загрузки могут проверяться по SHA-256
- сервер не должен принимать игрока без ModSync-handshake
- protected mod files не должны трогаться как обычный мусор

Но это не заменяет нормальную админскую дисциплину:
- не раздавай сомнительные файлы
- не смешивай production и тестовую сборку
- проверяй manifest после ручных изменений

## Частые вопросы

### Можно ли использовать только для resourcepacks/config?

Да. Просто отключи ненужные категории в `modsync.toml`.

### Можно ли раздавать shaderpacks?

Да, если `enable_shaderpacks_sync = true`.

### Можно ли раздавать client-only моды?

Да, через `optional_client`.

### Нужно ли ставить ModSync всем игрокам?

Да. Иначе сервер их не пропустит.

### Можно ли использовать в одиночной игре?

Нет, логика ModSync для handshake/sync в singleplayer должна пропускаться.

### Можно ли отключить проверку хэша после скачивания?

Да:

```toml
verify_hash_after_download = false
```

Но это менее безопасно.

### Почему после обновления модов нужен рестарт?

Потому что Forge-моды не применяются корректно "на лету".

## Быстрый чек-лист для админа

Перед запуском:

- ModSync стоит и на сервере, и на клиенте
- сервер на Forge 1.20.1
- Java 17
- `mods/` и `sync_repo/` заполнены правильно
- `server_http_port` доступен
- `public_http_base_url` пустой или корректный
- `logs/modsync.log` без ошибок

После изменений в сборке:

1. обновить файлы в `mods/` или `sync_repo/`
2. выполнить `/modsync manifest refresh`
3. проверить `/modsync status`
4. протестировать вход чистым клиентом с ModSync

---

## Быстрый чек-лист для игрока

- стоит Forge 1.20.1
- установлен ModSync
- достаточно места на диске
- нет блокировки HTTP запросов фаерволом/антивирусом
- после скачивания модов перезапустить игру, если ModSync попросил

## Итог

ModSync нужен для того, чтобы:
- не заставлять игроков вручную собирать клиент
- синхронизировать моды и клиентские файлы автоматически
- держать клиент ближе к состоянию сервера

Если его настроить аккуратно, это сильно упрощает поддержку сборки и снижает количество проблем при входе игроков.
